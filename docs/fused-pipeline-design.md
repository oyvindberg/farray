# Fused pipelines for FArray — design document

Status: design / not yet implemented. Audience: the engineer who will build this.
Prereq reading: skim `codegen/src/scala/farray/GenCores.scala` (the whole library is generated from there).

---

## 1. Goal

Make a chained pipeline

```scala
xs.fuse.map(f1).map(f2).filter(p).take(k).flatMap(g).map(h).toFArray
```

compile to **one fused traversal** of `xs` that produces **one** output collection, with:

- **no intermediate `FArray` per stage** (today each `.map`/`.filter` allocates a new backing array),
- **fully unboxed** stages (no boxing of primitives, no per-element virtual calls),
- **positional short-circuiting** (`take`/`find`/`head` bound the work so upstream stages run only as far as needed),
- and, as a later phase, **compute-for-survivors** (expensive independent columns evaluated only for elements that pass earlier filters).

This is loop fusion + a small compile-time query optimizer. It is implemented as a **Scala 3 macro** (multi-stage / "staging"). The macro is required — see §3.

## 2. Why — the evidence

Measured (`benchmarks/.../FusionBench.scala`, size 1000, `map(_+1).filter(_%2==0).map(_*2)` → `Array[Int]`):

| approach | ops/s | alloc B/op |
|---|---|---|
| eager chain (3 intermediate `IntArr`) | 805,788 | 14,080 |
| fused via **stored** specialized `Function1`s | 386,889 | 6,416 |
| **hand-inlined one-pass loop** | **929,504** | 6,416 |

Takeaways that drive the whole design:

1. **Fusion is a real win _only when fully inlined_.** The hand-inlined single loop beats the eager chain (+15%) and halves allocation. The "obvious" fused implementation — compose `Function1`s in a producer tower — is **2× slower than just chaining eager ops**, because FArray's eager combinators are already fully-inlined tight unboxed loops, so a per-element virtual-call tower costs more than the passes it removes.
2. **Therefore the stages must be inlined into one loop body — no stored lambdas, no closures.**
3. **`inline` alone can't do it**: Scala 3 `inline` lambdas cannot be stored in a field, so a *fluent* chain (each `.map` returning a value) cannot inline-compose across method boundaries. You need a **macro** that reads the whole chain as an AST and emits the loop.

So: macro, lowering to one nested unboxed loop, per primitive kind.

## 3. How the macro sees the pipeline

Fluent-macro pattern (AST-walking terminal):

- `xs.fuse` returns a marker type `Fuse[A]` (a thin wrapper around the source `FBase`).
- `Fuse[A]#map/filter/flatMap/take/...` are **ordinary methods** returning `Fuse[B]`. Their bodies are irrelevant (can be `???`/identity) — they exist only so the chain type-checks and so the lambdas live in the AST.
- The **terminals** are macros: `inline def toFArray: FArray[A] = ${ FuseMacro.run('this) }` (likewise `toArray`, `fold`, `foreach`, `find`, `count`, …).
- The terminal macro receives `'this` = the `Expr` of the whole receiver expression, i.e. the AST `xs.fuse.map(f1).filter(p)…`. It walks that AST, peeling `.map($f)`, `.filter($p)` … off the end back to `xs.fuse`, collecting the stage list and the lambda `Expr`s. Lambdas are extracted directly from the AST (no runtime function values ever exist).

This is the standard Scala 3 fluent-macro shape; the only subtlety is matching the desugared call nodes for each combinator.

## 4. What you're building on (existing FArray machinery — reuse it)

Everything is generated by `GenCores.scala`. Relevant facts:

- **`FBase` is a sealed tree.** Leaves: `IntArr/LongArr/DoubleArr/RefArr` (a backing array + length). Nodes: `Concat, IntAppend/…, IntPrepend/…, RangeNode, ReverseNode, SliceNode, IntPad/…, IntUpdated/…`. A pipeline source is any `FBase`.
- **Per-kind specialization.** Four "op kinds": `Int, Long, Double, Ref`. A `Repr[A]` typeclass (`IntRepr/LongRepr/DoubleRepr/RefRepr`) is summoned via `summonFrom`; at a concrete inline call site it resolves to exactly one kind. `RefRepr` no longer carries a `ClassTag` (reference backings are `Object[]`; reads cast the element `obj.asInstanceOf[A]`, valid for any backing). The fused macro must likewise specialize on the element kind(s).
- **Unboxed traversal already exists.** `dfsC<Kind>(node, consumer)` is a *non-inline* per-kind walk that calls `consumer.onLeaf(arr, len)` per contiguous run and `consumer.onOne(elem)` per single element. There is a leaf fast-path everywhere (`xs match { case leaf: IntArr => … fast … ; case _ => … }`). Use this for the **source loop and every flatMap inner loop** — do not invent a boxed iterator.
- **The one-pass `flatMap` is a working prototype of fused codegen.** See `flatMapOne` in `GenCores.scala`: it reads the source unboxed (leaf fast-path), applies an inline `f`, and appends results into a growable output sized from the first inner via `ensureCap<Kind>`. The fusion lowering generalizes exactly this shape. Reuse `ensureCap<Kind>` and the first-inner sizing trick.
- **Inline code-size is a hard constraint.** Chained inline combinators that each expand a full `dfs` blow past the JVM `HugeMethodLimit` (~8 KB bytecode) and run *interpreted* (measured 28× slowdowns). The fused macro avoids this naturally (it emits **one** loop nest, and the heavy traversal lives in the non-inline `dfsC`), but: keep the generated loop body lean, and push anything reusable into non-inline helpers.

## 5. Architecture: three layers

```
   surface API  ──►  Stage spine  ──►  Block IR (applicative DAG + nesting)  ──►  Expr (nested unboxed loops)
   xs.fuse.map…       [Map,Filter,        per loop-level: pure dataflow DAG          while-loops + builder
   .toFArray           FlatMap,Take…]      + guards + continuation                   (per kind)
```

- **Layer A — Stage spine.** The linear list of combinators as parsed from the AST. Enough for fusion + positional pushdown (the high-value core).
- **Layer B — Block IR.** Each loop level holds an *applicative block*: a pure dataflow DAG of `Term`s, an orderable set of guard `Term`s, and a continuation (`Emit` or `FlatMap` into a nested block). This is where projection/predicate pushdown and dead-binding sinking happen.
- **Layer C — Lowering.** Block → an `Expr[…]` of nested `while` loops over unboxed leaves, writing into a per-kind output builder.

Build A+C first (fusion works end-to-end). B is an optimization layer that rewrites between A and C.

## 6. Layer A — Stage spine

```scala
enum Stage:
  case Map     (f: Expr[Any => Any])
  case Filter  (p: Expr[Any => Boolean])
  case FlatMap (f: Expr[Any => FBase])        // monadic — opens nesting
  case Take(n: Expr[Int]); case Drop(n: Expr[Int])
  case Slice(from: Expr[Int], until: Expr[Int])
  case ZipWithIndex
  case Collect (pf: Expr[PartialFunction[Any, Any]])
  // terminals are separate (Emit kinds): ToFArray, ToArray, Foldl(z,op), Foreach(f),
  //   Find(p), Exists(p), Count, Head, …

case class Pipeline(source: Expr[FBase], elemKind: Kind, stages: List[Stage], terminal: Terminal)
```

The macro builds `Pipeline` by peeling the receiver AST. `elemKind` (and the kind after each stage) is derived from the static element types.

## 7. flatMap forces nesting — the loop spine

flatMap is the only combinator that is *not* 1-element-in/0-1-out: it iterates a sub-collection, so **everything downstream of it runs inside its inner loop**. The lowering target is a tree of loops, not a flat list:

```scala
enum Stmt:
  case Bind (name: Sym, value: Expr[?], body: Stmt)         // map:    val name = f(prev); <body>
  case Guard(pred: Expr[Boolean], body: Stmt)              // filter: if (pred) <body>
  case Loop (over: Expr[FBase], elem: Sym, body: Stmt)     // flatMap: foreach-unboxed over g(prev) { elem => <body> }
  case Emit (value: Expr[?])                               // builder += value
  case Done                                                // no-op tail
```

Build it by folding the stages into a continuation `K = Sym => Stmt`, **sink→source** (terminal first):

```
k0       = v => Emit(v)
Map(f)   : k => v => Bind(v', f(v), k(v'))
Filter(p): k => v => Guard(p(v), k(v))
FlatMap(g): k => v => Loop(g(v), w, k(w))         // ← inner loop; downstream k becomes its body
Take(n)  : k => v => Guard(count<n, { k(v); count+=1 })  // + sets the break flag when count==n
```

Wrap in the source loop: `Loop(source, x, kFull(x))`. flatMap is the *only* wrapper that emits a `Loop`; that is precisely why it (and only it) deepens nesting. Worked example `map(f).flatMap(g).filter(p).map(h)`:

```
Loop(source, x,
  Bind(y, f(x),
    Loop(g(y), w,
      Guard(p(w),
        Bind(z, h(w),
          Emit(z))))))
```

One source loop, one inner loop, no intermediate collection.

## 8. Layer B — the applicative DAG (the optimizer substrate)

The governing principle: **applicative = independent = freely movable; monadic (`flatMap`) = dependent = a fence.** Inside one loop level, replace opaque per-stage lambdas with a pure dataflow DAG so independence is *readable off the graph*.

```scala
enum Term:                                   // pure, hash-consed (CSE for free)
  case Src                                   // this level's input element
  case Idx                                   // its index (after ZipWithIndex)
  case Const(v: Expr[?])
  case Pure (fn: Expr[?], args: List[TermId])// n-ary apply; args are INDEPENDENT siblings   ← applicative
  case Tuple(parts: List[TermId])            // combine independent values                   ← applicative
  case Proj (of: TermId, i: Int)             // read one field

case class Block(
  graph:  Map[TermId, Term],
  guards: List[TermId],                      // filters as Boolean Terms (a set, not a list — reorderable)
  cont:   Cont)
enum Cont:
  case Emit(value: TermId)
  case FlatMapTo(sub: TermId, inner: Block)  // monadic edge to a deeper level
```

Reflecting a lambda into `Term`s: `Expr.betaReduce` the lambda applied to the current element symbol, then structurally translate the resulting tree — function applications → `Pure`, tuple literals → `Tuple`, `._n`/`apply` on a tuple → `Proj`, literals → `Const`. Anything you can't safely analyze becomes a single atomic `Pure(<whole lambda>, [input])` (correct, just not splittable — graceful degradation).

### 8.1 Optimization passes (all on a `Block`, none crossing a `FlatMapTo`)

1. **Proj-simplify:** `Proj(Tuple(xs), i) ⟶ xs(i)`. Tuples evaporate; independence between former fields becomes explicit.
2. **CSE:** hash-consing makes identical `Pure` subtrees one node (e.g. `x => (f(x), f(x)+1)` computes `f(x)` once).
3. **DCE:** any `Tuple` part / `Term` not reachable from a live guard or the continuation is deleted — *compute only the columns you use*.
4. **Sink (partial dead-code / lazy code motion):** schedule each `Term` immediately before its earliest live use. If that point is *after* a guard, the term is computed **only for survivors**; if it's inside `FlatMapTo.inner`, only when that loop runs.
5. **Guard reorder:** guards are a set; emit cheapest / most-selective first (heuristic by node count now; cost/selectivity annotations later).

Worked example `xs.map(x => (cheap(x), expensive(x))).filter(_._1>0).map(_._2)`:

```
t1=Pure(cheap,[Src])  t2=Pure(expensive,[Src])  t3=Tuple([t1,t2])
guard = Pure(_>0,[Proj(t3,0)])      --Proj-simplify-->  guard on t1
emit  = Proj(t3,1)                  --Proj-simplify-->  emit t2
sink t2 past the guard  ⟹  a=cheap(x); if(a>0){ b=expensive(x); emit b }
```

`expensive` runs only for survivors — predicate + projection pushdown, falling out of generic graph rewrites because the applicative structure exposed the independence.

### 8.2 Two flavors of independence
- **Per-element** (above): `Tuple`/`Pure` → column pruning + sinking.
- **Stream-level:** `zip`/`map2`/`mapN` combine independent *sources* → one lockstep loop, no intermediate `(A,B)` array; each side may carry its own fused sub-pipeline. (`flatMap` cannot — it's monadic.) Model a zip as a loop level with two `Src`s advanced together.

## 9. Layer C — lowering to `Expr`

`Block`/`Stmt` → nested unboxed loops, specialized per kind:

- **Source / inner loops:** lower `Loop(over, elem, body)` to the existing unboxed traversal of `over`: leaf fast-path (`case leaf: KArr => { val a=leaf.arr; var i=0; while(i<n){ val elem = a(i) /* + element cast for Ref */; <body>; i+=1 } }`) plus the `dfsC<Kind>` consumer for non-leaf nodes. Reuse what `dfsC`/`flatMapOne` already do; do **not** box.
- **`Bind`** → `val name = <value>` (the inlined map expression). **`Guard`** → `if (<pred>) { <body> }`. **`Emit`** → `out = ensureCap<K>(out, off+1); out(off) = <value>; off += 1`.
- **Output sizing:** carry `boundedSize: Option[Expr[Int]]` on the pipeline. Pure-`map`/positional-only chains have an exact size ⟹ preallocate `new Array[…](size)` and skip the builder. Anything with `filter`/`flatMap` ⟹ growable output via `ensureCap<K>` (size the initial array from `source.length`, like `flatMapOne`). Trim at the end if `off < out.length`.
- **Break for `take`/`find`/`exists`/`head`:** thread a single `var done = false`; every emitted `Loop` adds `&& !done` to its condition; the short-circuiting stage sets `done = true`. No exceptions, fully unboxed, escapes all nesting levels.
- **Result:** wrap the final array in the right leaf (`new KArr(out, off)`), returned as the opaque `FArray`.

## 10. Semantics & safety (read this twice)

The optimizer **changes how often and in what order user functions run** — this is a deliberate, query-optimizer-style choice and must be documented in the public API.

- **Purity assumption.** Sinking, DCE, CSE, guard-reorder, and positional pushdown are valid only for **pure** stage functions (no side effects, referentially transparent). With a pure `f`, results are identical to a naive pipeline; only the *count/order* of `f`-evaluations changes. This differs from `scala.collection`, where `map` runs `f` for every element. **Default stance:** treat every reflected lambda as atomic & unmovable unless it is provably a pure projection/arithmetic tree (no `var`, no I/O, no opaque effectful calls). Fusion (one pass) and positional pushdown (fewer source elements visited) are *always* safe regardless of purity, because they don't reorder or duplicate `f` relative to the elements it would have seen — they only stop early. Only the **sink/DCE/reorder** passes need purity; gate them behind the analyzability check, and consider an opt-in (`xs.fuse(assumePure = true)…` or a `@pure` marker) before enabling aggressive moves.
- **Evaluation-count visibility.** Even pure-but-expensive `f` benefits from sinking, but a user counting calls would see fewer. Document it.
- **`flatMap` is a hard fence.** No applicative rewrite crosses a `FlatMapTo`. Effects ordering across it is preserved.
- **Exceptions / partiality.** If a sunk computation can throw, sinking it past a filter changes whether it throws for filtered elements. Same purity gate covers this.

## 11. Phasing (suggested PR order)

1. **Spine + fusion + lowering (Layer A→C), `map`/`filter`/`take`/`drop`/terminals `toFArray`/`toArray`/`foreach`/`fold`.** End-to-end one fused loop, per kind. Positional pushdown (`take`/`drop` as loop bounds + break flag) is free here. Ship and benchmark vs eager — target the 929k/−50%-alloc hand-fused number.
2. **`flatMap` + the nested `Loop` lowering** + `find`/`exists`/`head`/`count` terminals (break across nesting).
3. **`zipWithIndex` + index-only predicate pushdown** (positional, no value needed).
4. **Layer B applicative DAG**: reflect lambdas → `Term`s, Proj-simplify + CSE + DCE + sink (the compute-for-survivors win). Gate on purity.
5. **Guard reorder + stream-level `zip`/`map2`.**

Each phase is independently shippable and benchmarkable.

## 12. Risks / open questions

- **Lambda reflection robustness.** `betaReduce` + structural translation must degrade gracefully to atomic nodes on anything unexpected; never miscompile. Heavy test burden.
- **Type tracking through the chain.** Need the element kind after each stage (for specialization) and tuple field types — derivable from the call AST's static types, but fiddly with generics.
- **Purity gate.** Conservative analysis (whitelist: arithmetic, field/tuple access, known-pure stdlib) vs. an opt-in. Start opt-in; Phase 1–3 don't need it at all.
- **Compile-time cost.** A query optimizer in a macro adds compile time; keep passes linear and the IR small.
- **API ergonomics.** `fuse` / terminal naming, interaction with the existing eager methods (don't shadow), and what happens to a `Fuse[A]` that's never terminated (should be a compile error or a no-op materialize).
- **Debuggability.** Provide a `-Dfarray.fuse.dump` that prints the lowered `Stmt`/loop for a pipeline, so users can see what was generated.

## 13. Test strategy

- **Correctness:** for every combination of stages over Int/Long/Double/Ref, assert the fused result `==` the eager result (and `== List`'s result), including empties, single elements, all-filtered-out, `take(0)`, `take(>size)`, nested `flatMap`.
- **Semantics:** a counter-instrumented (pure) `f` to assert sink actually reduces call counts in the survivor case (Phase 4), and that fusion/positional never *increase* them.
- **No-blowup:** a long chain (8+ stages) must stay JIT-compiled (regression test à la `MapChainBenchmark`) — confirm the macro emits one method under the size limit.
- **Bench:** extend `FusionBench` to cover map/filter/take/flatMap and the survivor-sink case; track vs eager and vs the hand-fused ceiling.
